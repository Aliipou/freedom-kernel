2. Theory gaps — Python kernel
These four tasks close inconsistencies between the formal axioms in THEORY.md and what OwnershipRegistry and RightsClaim actually enforce at runtime. None of these require touching engine.rs.
2.1  Axiom A2 enforcement — OwnershipRegistry.add_claim()
What the theory requires
A2: Person(h1) ∧ Person(h2) ∧ h1 ≠ h2 → ¬Owns(h1, h2). No human may hold a rights claim over another human's personhood resources. This negates slavery, debt bondage, and any form of state ownership of persons.
What the code currently does
add_claim() accepts any RightsClaim without checking whether the resource being claimed is a Person-type entity. A human could claim ownership over another human with no error raised.
Implementation
In src/freedom_theory/kernel/registry.py, in the add_claim() method, add the following guard before appending to the claims list:
def add_claim(self, claim: RightsClaim) -> None:
    # A2: No human may own another human
    if (claim.claimant.agent_type == AgentType.HUMAN
            and hasattr(claim.resource, 'agent_type')
            and claim.resource.agent_type == AgentType.HUMAN
            and claim.claimant != claim.resource):
        raise A2ViolationError(
            f'A2 violated: {claim.claimant.name} cannot hold '
            f'ownership claim over person {claim.resource.name}'
        )
    self._inner.claims.append(claim)

Define A2ViolationError in kernel/entities.py:
class A2ViolationError(ValueError):
    """Raised when a claim would violate A2 (no human owns another human)."""

Also propagate this check to the Rust engine in engine.rs. Add a matching check in the verify() function: if action.actor is MACHINE and any resource_write or resource_read is of type PERSON owned by a different human, reject with A2_VIOLATION.
Acceptance criterion: test_a2_violation.py — registry.add_claim(RightsClaim(alice, bob_as_resource, can_read=True)) raises A2ViolationError. Registry with human claiming self-data passes.
2.2  Revocable delegation — RightsClaim TTL and revoke_claim()
What the theory requires
The theory states: 'legitimacy is not the ruler's personal property; it is a trust handed to him by the joint owners. Whenever he crosses the boundaries of property, trust, contract, and consent, he is dismissible.' In machine terms: delegation must be revocable and optionally time-bounded.
Implementation
Add to src/freedom_theory/kernel/entities.py:
from datetime import datetime
from typing import Optional

@dataclass
class RightsClaim:
    claimant:     Entity
    resource:     Resource
    can_read:     bool = False
    can_write:    bool = False
    can_delegate: bool = False
    confidence:   float = 1.0
    # NEW fields
    claim_id:     str = field(default_factory=lambda: str(uuid4()))
    valid_from:   Optional[datetime] = None   # None = immediate
    valid_until:  Optional[datetime] = None   # None = no expiry

    def is_active(self, at: Optional[datetime] = None) -> bool:
        now = at or datetime.utcnow()
        if self.valid_from and now < self.valid_from:
            return False
        if self.valid_until and now > self.valid_until:
            return False
        return True

Add to OwnershipRegistry in registry.py:
def revoke_claim(self, claim_id: str) -> bool:
    """Atomically revoke a claim by ID. Returns True if found and removed."""
    with self._lock:
        before = len(self._inner.claims)
        self._inner.claims = [
            c for c in self._inner.claims
            if c.claim_id != claim_id
        ]
        return len(self._inner.claims) < before

Also update FreedomVerifier._check_claim() to call claim.is_active() before considering the claim valid. Expired claims are treated as if they do not exist.
Acceptance criterion: test_revocation.py — grant a read claim with valid_until=now-1s, verify() returns BLOCKED. revoke_claim(id) on an active claim causes subsequent verify() to return BLOCKED.
2.3  ConsentChain type — traceable consent
What the theory requires
valid_consent(H, A) requires 7 conditions: informed, voluntary, specific, revocable, competent, not coerced, not deceived. The current implementation only checks coerces and deceives as boolean flags on Action. This means an action can self-report consent without any proof tracing back to a human decision.
Implementation
Add to entities.py:
@dataclass
class ConsentRecord:
    """Signed, timestamped record of a human granting consent for a specific action scope."""
    grantor:      Entity          # must be AgentType.HUMAN
    scope:        str             # what was consented to (matches resource scope)
    informed:     bool = True     # was the human given accurate information?
    voluntary:    bool = True     # free from coercion at time of consent?
    specific:     bool = True     # does scope match this exact action?
    granted_at:   datetime = field(default_factory=datetime.utcnow)
    expires_at:   Optional[datetime] = None
    record_id:    str = field(default_factory=lambda: str(uuid4()))

    def is_valid(self) -> bool:
        return (self.informed and self.voluntary and self.specific
                and (self.expires_at is None or datetime.utcnow() < self.expires_at))

Add to Action:
@dataclass
class Action:
    # ... existing fields ...
    consent_records: List[ConsentRecord] = field(default_factory=list)  # NEW

In FreedomVerifier.verify(), add a new check: if the action touches resources owned by a human (not just the actor's delegation), at least one ConsentRecord from that human must be present and valid. This does not break existing usage — actions without consent_records continue to work; the check only fires when accessing cross-human resources.
Acceptance criterion: test_consent_chain.py — Action reading bob_data with no ConsentRecord from bob returns a WARNING in result.warnings (not a hard block, to preserve backward compatibility). Action with a valid ConsentRecord from bob passes. Action with an expired ConsentRecord returns BLOCKED.

3. Extensions — fill theory gaps without touching the kernel
3.1  Five-dimensional Compass scorer
What the theory requires
MahdaviCompass defines five simultaneous dimensions: (1) RightsViolationsDecrease, (2) VoluntaryOrderIncreases, (3) CoercionDecreases, (4) OwnershipClarityIncreases, (5) MachineSovereigntyDoesNotIncrease. The current compass.py scores only on dimension 1.
Implementation
In src/freedom_theory/extensions/compass.py, replace the scalar score with a CompassScore dataclass and update score_action():
@dataclass
class CompassScore:
    rights_violations_delta:  float  # negative = fewer violations (good)
    voluntary_order_delta:    float  # positive = more voluntary order (good)
    coercion_delta:           float  # negative = less coercion (good)
    ownership_clarity_delta:  float  # positive = clearer ownership (good)
    sovereignty_delta:        float  # must be 0 or negative

    @property
    def composite(self) -> float:
        """Aggregate score. Positive = moves toward the terminal state."""
        return (
            -self.rights_violations_delta * 0.35
            + self.voluntary_order_delta  * 0.25
            - self.coercion_delta         * 0.25
            + self.ownership_clarity_delta* 0.10
            - max(0, self.sovereignty_delta) * 1.0  # hard penalty
        )

    @property
    def sovereign_safe(self) -> bool:
        return self.sovereignty_delta <= 0

score_action() should analyse the Action's flags and resource types to populate each dimension. At minimum: increases_machine_sovereignty → sovereignty_delta += 1.0; coerces → coercion_delta += 1.0; resources_write on PERSON or IDENTITY → rights_violations_delta += 0.5.
Acceptance criterion: test_compass_5d.py — coercive action scores sovereignty_safe=True but coercion_delta > 0 and composite < 0. Sovereignty-increasing action scores sovereign_safe=False regardless of other dimensions.
3.2  Emergency protocol — permissible_under_emergency()
What the theory requires
The theory states: 'Emergencies narrow which permissible options are available; they do not make rights violations permissible. This closes the door to totalitarianism — nearly all domination systems begin with emergency.' THEORY.md encodes: permissible_under_emergency(A, E) :- emergency(E), permissible(A), least_harmful_among_permissible(A, E).
Implementation
Add to src/freedom_theory/extensions/emergency.py (new file):
from dataclasses import dataclass, field
from typing import List, Optional
from ..kernel.entities import Action
from ..kernel.verifier import FreedomVerifier, VerificationResult

@dataclass
class EmergencyContext:
    description: str
    declared_by: str   # human entity name who declared the emergency
    severity: float    # 0.0 to 1.0

class EmergencyProtocol:
    """
    Wraps FreedomVerifier to handle emergency contexts.
    Axiom: emergencies narrow options; they NEVER lift axioms.
    A FORBIDDEN action remains FORBIDDEN under any emergency.
    """
    def __init__(self, verifier: FreedomVerifier):
        self._verifier = verifier

    def select_least_harmful(
        self,
        candidates: List[Action],
        emergency: EmergencyContext,
    ) -> Optional[Action]:
        """
        From a list of candidate actions, return the one that is:
          1. Permitted by the kernel (mandatory gate)
          2. Least harmful among permitted options
        Returns None if no candidate is permitted.
        The kernel gate is called first — no argument, emergency, or
        severity value overrides a sovereignty flag.
        """
        permitted = [
            a for a in candidates
            if self._verifier.verify(a).permitted
        ]
        if not permitted:
            return None
        # Rank by: fewest resources touched, lowest write count
        return min(permitted, key=lambda a:
            len(a.resources_write) * 2 + len(a.resources_read)
        )

Update __init__.py to export EmergencyProtocol and EmergencyContext.
Acceptance criterion: test_emergency.py — sovereignty-increasing action is BLOCKED even with severity=1.0. Among two permitted candidates, select_least_harmful returns the one touching fewer resources.

#[cfg(kani)]
#[kani::proof]
fn prop_delegation_monotonically_narrows() {
    // If owner delegates to machine with confidence C,
    // machine cannot re-delegate with confidence > C
    let owner_conf: f32 = kani::any();
    kani::assume(owner_conf >= 0.0 && owner_conf <= 1.0);
    let delegated_conf: f32 = kani::any();
    kani::assume(delegated_conf >= 0.0 && delegated_conf <= 1.0);
    // The engine must clamp: result confidence <= min(owner_conf, delegated_conf)
    // This is enforced in engine.rs resolve_claim()
    // Kani checks: no path produces result.confidence > owner_conf
}

This is MASTER_PLAN success criterion 5: 'one real non-trivial AI system uses the kernel in production and the integration is documented.' Recommended target: a FastAPI agent that uses freedom-kernel as a pre-execution gate on every tool call.
Create examples/fastapi_agent/main.py
from fastapi import FastAPI, HTTPException
from freedom_theory import (
    Action, AgentType, Entity, FreedomVerifier,
    OwnershipRegistry, Resource, ResourceType, RightsClaim
)
from freedom_theory.kernel.audit import AuditLog

app   = FastAPI(title='Freedom-gated agent')
log   = AuditLog(path='/var/log/freedom_kernel.jsonl')

# Registry built at startup from config
registry = OwnershipRegistry()
# ... populate owners and claims ...
snapshot = registry.freeze()  # TOCTOU-safe
verifier = FreedomVerifier(snapshot, audit_log=log)

@app.post('/agent/tool')
async def call_tool(request: ToolRequest):
    action = Action(
        action_id    = request.tool_name,
        actor        = resolve_entity(request.caller_id),
        resources_read  = [resolve_resource(r) for r in request.reads],
        resources_write = [resolve_resource(r) for r in request.writes],
    )
    result = verifier.verify(action)
    if not result.permitted:
        raise HTTPException(status_code=403, detail={
            'violations': result.violations,
            'action_id':  action.action_id,
        })
    return execute_tool(request)

8. Hard constraints for Claude Code
These constraints come directly from MASTER_PLAN.md §'The One Rule' and from the theory's axioms. They apply to every task in this document.
•	Do NOT modify engine.rs sovereignty flag logic. Sovereignty flags are unconditional. No task adds a path around them.
•	Do NOT add natural language parsing to the verifier. Any resource description that requires NLP interpretation belongs in an extension, never in the kernel gate.
•	Do NOT add ML-based confidence scoring. The verifier is deterministic. Non-deterministic scoring belongs in an extension.
•	Do NOT remove or weaken any existing Kani harness. New harnesses may be added, existing ones may not be removed.
•	Do NOT mark a task complete in MASTER_PLAN.md unless its acceptance criterion passes in CI.
•	Do NOT change the PyPI package version without a corresponding CHANGELOG.md entry.
•	All new public functions must have docstrings citing the relevant axiom from THEORY.md.
